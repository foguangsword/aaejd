# 第3章：面向开发者的提示工程——为什么提示在AI赋能应用中至关重要

> **原文：** Applied AI Enterprise Java — Chapter 3: Prompts for Developers: Why Prompts Matter in AI-Infused Applications

---

提示（Prompt）是与LLM交互的主要机制。它们定义了AI系统如何响应，影响着生成内容的质量、相关性和可靠性。对于构建AI赋能应用的Java开发者来说，理解提示设计是最重要的技能之一。一个结构良好的提示可以减少幻觉（hallucination）、提高一致性并优化性能，而无需对模型进行微调。关于编写有效提示和最佳技术，有很多可用的建议，例如OpenAI提示工程指南，或James Phoenix和Mike Taylor合著的《Prompt Engineering for Generative AI》（O'Reilly出版）。请将本章视为一个简要概述和你学习旅程的起点。

## 提示的类型

提示根据其来源和引导模型的方式而有所不同。主要类型包括用户提示、系统提示和上下文提示。让我们逐一来看。

### 用户提示：来自用户的直接输入

用户提示是由最终用户提供的原始输入。这些通常是非结构化的，需要预处理或上下文增强以确保准确的响应。示例如下：

```java
String userPrompt = "What is the capital of France?";
```

有效处理用户提示需要输入清理、意图识别和上下文增强。我们将在第4章更详细地讨论这一点。

### 系统提示：引导模型行为的指令

系统提示定义了模型在会话中的行为方式。这些通常在交互开始时设置，并对用户保持隐藏。它们可用于建立语调、强制执行约束或引导模型的响应。示例如下：

```java
String systemPrompt = "You are a helpful AI assistant\n that provides concise and factual responses.";
```

系统提示定义了LLM在应用中的边界。它们也可以用于强制执行某些输出或包含工具调用指令。

### 上下文提示：预填充或动态生成的输入

上下文提示包含添加到提示中的背景信息、历史交互或领域特定知识，以改善响应。这些可以基于用户历史或外部数据动态生成。这也是一种将会话记忆注入对话的有效方式。我们将在第4章更多介绍其架构方面的内容。以下是一个上下文提示的示例：

```java
String context = "User previously asked about European capitals.";
String fullPrompt = context + " " + userPrompt;
```

上下文提示增强了响应的相关性，特别是在多轮对话中，并充当事实上的记忆，帮助LLM保持对话的连贯性。

## 编写有效提示的原则

**具体性和结构性**对于有效的提示工程至关重要。通过精确并在逻辑上组织你的提示，你可以显著提高从LLM获得的响应的质量和相关性。投入时间精心设计结构良好且具体的提示是充分利用这些强大工具的关键步骤。以下分别是过于宽泛和过于具体的提示的示例：

```java
String vaguePrompt = "Tell me about Java.";  // 太宽泛
String specificPrompt = "Explain Java's garbage collection \n
    mechanisms in one paragraph.";  // 好得多
```

`vaguePrompt`是如此开放式，以至于LLM可能以任何与Java相关的内容来回应——也许是它的历史、它的用途或它的语法。而`specificPrompt`则清楚地说明了所需的信息，甚至指定了期望的长度（一个段落）。

为LLM精心设计好的提示对于获得你想要的响应非常重要。常见错误会使这一点变得困难。一个问题是**过于模糊**，这会导致不清晰或笼统的结果。给模型**过多信息**也会使其困惑。牢记**提示长度**也很重要，因为非常长的提示可能会被截断。最后，你应该**测试和修改**你的提示以使它们变得更好。

## 提示技术

各种提示技术提供了与模型交互的结构化方式，从直接的指令和示例到增强推理或引入外部知识的更复杂方法。

### 零样本提示：无上下文地提问

在零样本提示中，你要求模型做某事而不给它任何示例。模型使用它在训练期间学到的内容来理解和完成请求。这种方法就像向一个对某个主题非常了解的人提问而不给他们任何背景信息。零样本提示完全依赖模型的预训练知识。一个简单的示例如下：

```java
String zeroShotPrompt = "Define polymorphism in object-oriented programming.";
```

虽然零样本提示可以很好地工作，但它有限制：准确率可能因任务复杂度而异；模糊的提示可能被误解；模型可能对全新的任务有困难。零样本提示适合快速测试LLM的能力。它是一个好的起点，但对于更复杂的任务，可能需要其他方法。

### 少样本提示：提供示例以引导响应

通过少样本提示，你提供一些你想让模型执行的任务的示例。这些示例展示了期望的输入-输出关系，并帮助引导模型生成正确类型的响应。这种方法就像在要求某人自己做某事之前，先给他们看几个如何做的例子。在Java中使用示例可以像这样：

```java
String fewShotPrompt = "Translate the following phrases to Spanish:\n\n" +
    "English: Hello\n" +
    "Spanish: Hola\n\n" +           // 示例1
    "English: Good morning\n" +
    "Spanish: Buenos días\n\n" +    // 示例2
    "English: How are you?\n" +
    "Spanish: ";  // LLM完成这部分
```

在这个示例中，你给了LLM两个英译西班牙语的例子。这帮助模型理解你想要"How are you?"的西班牙语翻译。模型更有可能给出正确的翻译（"¿Cómo estás?"），而不是如果你使用零样本提示。通过看到一些示例，LLM可以更好地理解你希望它遵循的模式或规则。它可以从这些示例中泛化，并将学到的模式应用于新的、未见过的输入。这对具有特定期望输出格式或稍微模糊的任务特别有帮助。

### 思维链提示：鼓励逐步推理

思维链（CoT）提示旨在通过在给出最终答案之前显式生成一系列中间步骤或"思维链"，来帮助LLM执行复杂的推理。这种方法就像要求某人在数学问题上展示他们的解题过程。你不是只得到最终结果，而是看到导致它的逐步推理。这对于像下面这样的文字题非常有效：

```
问题：Roger有5个网球。他又买了2罐网球。每罐有3个网球。他现在有多少个网球？

思考链：
Roger开始时有5个网球。
他买了2罐，每罐3个网球，所以他买了 2 × 3 = 6 个网球。
总共，他有 5 + 6 = 11 个网球。

答案：11
```

你不仅仅给LLM问题并要求答案。你正在向模型展示如何通过分解步骤来解决问题。如果你随后给LLM一个类似的问题，它更有可能生成自己的思维链并得出正确答案。LLM从它们训练的海量数据集中学到了很多关于推理的知识。然而，它们在回答问题时并不总是显式地使用这种推理能力。CoT提示通过提供如何逐步思考的示例，鼓励它们激活和利用其推理能力。

### 自一致性：通过生成多个响应提高准确率

自一致性是提高LLM响应准确率的一种方法，特别是在推理任务上。核心思想很简单：不依赖单一生成的响应，而是生成多个响应，然后选择最一致的一个。这利用了这样一个想法：虽然LLM在其推理中可能偶尔出现错误，但正确答案更有可能在多次尝试中一致地出现。

LLM是概率模型。它们并不总是对相同的输入产生相同的输出。有时它们会犯错误，特别是在复杂的推理任务上。然而，自一致性背后的假设是，即使LLM偶尔犯错误，正确答案也更有可能被反复生成。通过生成多个响应，你增加了捕获正确答案并过滤掉错误答案的机会。

### 指令提示：显式指导模型

指令提示是一种直接的方法。它涉及给模型明确的指令，说明你想让它做什么。你不是依赖隐含的线索或间接的建议，而是直接告诉LLM要执行的任务和你期望的输出类型。这种方法就像给某人清晰具体的指示。一个非常明确的提示如下所示：

```
TRANSLATE: English to Spanish
TEXT: "Hello, world!"
```

指令提示通常与其他提示方法结合使用。你可以在少样本学习设置中使用指令提示，其中示例也包含清晰的指令。它还可以与思维链提示结合，通过指示模型在提供最终答案之前"逐步展示其推理过程"。

### 检索增强生成：用外部数据增强提示

RAG本质上是上下文提示的一个版本。它不是使用任何文本上下文，而是在回答问题之前使用外部来源（如数据库或文档）来获取更当前或更具体的信息。从技术上讲，这更像是一种架构方法而非提示技术。我们将在第5章更详细地研究其工作原理，并逐步解开其复杂性。

## 高级策略

在基础技术的基础上，本节涵盖高级提示策略，用于创建与语言模型之间更加动态、可靠和优化的交互。你将在第6章和第7章再次看到这些通用示例中的许多。

### 构建动态提示：结合静态和生成的输入

这种方法通过将固定文本与系统其他部分生成的信息相结合来动态构建提示。例如，你可能有一个用于总结产品的模板提示，但具体的产品细节（名称、描述、价格）是从数据库中提取的，并在发送给模型之前插入到提示中。这允许灵活且上下文感知的提示。你可以使用Java的String特性来动态添加内容，如下例所示：

```java
String productTemplate = "Summarize product:\n
    Name: {productName}\n
    Description: {productDescription}";
String productName = "Wireless Headphones";
String productDescription = "Noise-canceling, Bluetooth 5.0";
String dynamicPrompt = productTemplate
    .replace("{productName}", productName)
    .replace("{productDescription}", productDescription);
// 'dynamicPrompt' 可以发送给模型。
```

使用代码从模板和变量数据动态构建提示，以提高灵活性和上下文感知能力。

### 使用提示链维护上下文

如果你需要跨与模型的多次交互维护上下文，你可以使用**提示链**。不要孤立地对待每个提示，而是将它们链接在一起。一个提示的输出成为下一个提示输入的一部分。这对于多步骤任务很有用，比如构建故事或回答需要多条信息的复杂问题。想象一次对话，你仍然记得开头，而不仅仅是回复下一个问题。将其想象为一条链——一个提示产生一个响应，然后将其插入到下一个提示中：

```java
String initialPrompt = "List the ingredients in a Margherita pizza.";
String firstResponse = getLLMResponse(initialPrompt);  // 假设的LLM调用
String followUpPrompt = "How do I make a pizza with these ingredients:\n"
    + firstResponse;
String finalResponse = getLLMResponse(followUpPrompt);  // 假设的LLM调用
```

提示链通过将先前的模型响应纳入后续提示，实现了多步骤问题解决或非常小型的对话记忆。

### 使用护栏和验证确保更安全的输出

你会发现对护栏（guardrails）和验证（validations）这两个术语有很多定义，但它们基本上都意味着同一件事：确保模型的输出是**安全**和**可靠**的。护栏可能过滤或拒绝包含有害内容的输出。验证可能检查输出是否符合某种格式或逻辑。例如，如果你要求模型生成代码，你可能会验证代码是否正确编译。我们将在第7章研究更具体的实现。你可以像下面这个示例一样将它们手动构建到你的代码中，或者使用库的特性，如Quarkus或LangChain4j为某些模型提供的功能：

```java
String llmR = getLLMResponse("Write a short story.");  // 假设的LLM调用
// 简单护栏：检查有害内容
if (llmR.contains("violent") || llmR.contains("hate")) {
    System.out.println("Response flagged for inappropriate content.");
} else {
    // 验证（示例：检查长度）
    if (llmR.length() > 500) {
        System.out.println("Response too long. Truncating.");
        llmR = llmR.substring(0, 500);
    }
    System.out.println(llmR);
}
```

对模型输出应用护栏和验证，以强制执行安全标准并验证是否符合预期。

### 利用API进行提示定制

模型供应商通常提供允许你定制提示过程的API。这些API可能允许你设置控制模型行为的超参数，或者可能提供管理和组织提示的工具。它们的能力取决于所使用的API和所暴露的功能。

### 性能与成本的优化

生成更长的响应或进行大量API调用会在使用费用或资源方面累积成本。因此，同时优化**性能**（获得良好结果）和**成本**（最小化开支）非常重要。这可能涉及使用更短的提示、缓存常见响应，或为不太关键的任务选择更便宜的LLM。

### 调试提示：排查糟糕的响应

调试提示涉及弄清楚为什么LLM给出了糟糕的响应，然后修改提示来修复问题。这通常需要仔细分析提示和LLM的输出以定位问题所在。这个过程就像调试代码，但你不是在调试代码，而是在调试你的问题。

掌握提示技术让我们能够直接控制模型交互，但AI赋能的应用需要更多技术组件。让我们看看你通常会遇到的支撑技术，如嵌入、向量数据库、缓存、Agent以及促进更复杂解决方案的框架。

虽然提示工程提供了一个有用的起点，但在现实世界的应用中，当我们遇到上下文窗口限制或意识到静态输入本身不足够时，它的局限性很快变得明显。特别是，对**动态上下文组装和管理**的需求变得清晰。我们称这为**上下文工程（Context Engineering）**，这是一种面向用户的代表用户进行内存管理的系统级方法，它超越了处理单个提示，而是考虑围绕模型的整个信息流。我们将在第4章详细研究上下文和记忆。

## 工具使用与函数调用

LLM可以通过与外部系统、API和工具交互来超越简单的文本生成，以增强其能力。当一个模型声称支持**工具使用（Tool Use）**或**函数调用（Function Calling）**时，这意味着它已经被专门设计或微调以解释、生成和执行结构化的函数调用，而不仅仅是用原始文本响应。这些特性使LLM在从API检索数据、运行数据库查询、执行计算或触发自动化工作流时更加可操作和有用。

**工具使用**指的是LLM决定何时以及如何使用外部资源来完成任务的能力。模型可以调用预定义的工具、检索必要的信息并将其纳入其响应中，而不是直接回答问题。LLM启用工具使用的一种方式是通过系统提示，模型依赖嵌入在系统消息中的预定义指令来确定何时以及如何调用外部工具。这些提示引导模型的行为，帮助它识别何时需要API调用或外部检索而不是直接响应。一个示例系统提示可以如下：

```
In this environment you have access to a set of tools you can use to answer
the user's question.
{{ FORMATTING INSTRUCTIONS }}
Lists and objects should use JSON format. The output is expected to be valid XML.
Here are the functions available in JSONSchema format:
{{ TOOL DEFINITIONS IN JSON SCHEMA }}
{{ USER SYSTEM PROMPT }}
{{ TOOL CONFIGURATION }}
```

通过精心设计系统提示，你可以引导模型以结构化的方式决定何时用哪个答案来响应。这只是提示的一种方式。

对于更高级的工具使用，可以使用**自定义模型适配器**为现有模型添加一个微调层。这种方法需要额外的训练或特定的适配器模型，但允许你将模型适配到特定用例甚至特定领域。微调的适配器不依赖通用的指令，而是提高模型检测何时应调用外部函数以及生成符合给定服务要求的精确API请求的能力。我们将在第5章讨论工具和函数调用的架构方面。

## 上下文工程——新一代提示工程

随着AI应用从原型发展到生产系统，**上下文工程（Context Engineering）**引入了一系列远超仅创建提示的新挑战。虽然LLM的上下文窗口可以容纳大量信息，但我们组装、管理和使用上下文的方式对应用的可靠性和性能至关重要。

上下文工程是将LLM视为一个**可编程的子系统**，其中提示只是在正确的时间传递正确的信息，而不是交互的唯一焦点。这种转变需要我们以更全面的方式来设计AI赋能系统中的信息流。

随着上下文窗口的增长，一个新问题出现了——**大海捞针效应（needle-in-a-haystack effect）**。充满聊天记录、检索到的文档、用户偏好和工具结果的大载荷可能会稀释任何单一信息的相关性。这使得模型更难识别和推理最重要的输入。为了缓解这个问题，我们需要实现智能的上下文剪枝策略。像按语义相关性对检索内容进行重排序、块评分或摘要这样的技术成为首选工具。系统不应该是盲目地将所有内容注入提示，而是应该仔细选择和塑造进入模型工作记忆的内容，最大化**相关性/Token比率**。

此外，LLM处理的每个token都伴随着成本和延迟。添加更多上下文直接增加了每个请求的运行时和计费占用。开发者必须精心设计上下文组装流水线，只包含必要的内容，压缩冗长的信息，并在可能的情况下缓存重复的查询。优化上下文元素的结构和顺序也可以通过引导注意力在提示早期关注最重要的细节来改善模型性能。

当上下文从私人和专有来源动态组装时，**访问控制**成为另一个关注点。RAG流水线必须确保只有用户被授权查看的文档才被检索并呈现给模型。这不是可以委托给LLM的责任，它必须在检索和编排层强制执行。细粒度的访问控制、用户上下文范围和多租户数据隔离是这里相关的设计考虑因素。如果没有这些控制，存在将敏感信息暴露给错误用户或允许模型基于它本不应访问的上下文产生幻觉的真实风险。

上下文工程，就其核心而言，是从将LLM视为不透明到将其视为**可编程子系统**的转变。在这个模型中，提示工程和推理只是更广泛信息流的一部分。不要将提示设计视为AI赋能应用中的核心。开发者需要控制围绕模型的整个信息生命周期。

## 为上下文设计内存与存储

上下文工程不仅仅是关于架构模式和系统设计。它在很大程度上依赖正确的技术组件来支持动态、可扩展和高效的上下文构建。分层的数据基础设施对于管理信息在推理时如何存储、检索和传递给模型是必需的。这些组件可以按数据访问的频率和需要多快可用来分类，遵循类似于传统计算系统的内存层次结构。

### 使用内存缓存快速访问

对于需要在短时间内重复访问的上下文元素（如最近的聊天记录、系统提示或跨请求共享的检索结果），**内存缓存**提供最快的响应。缓存在无状态环境中特别有用，因为相同的上下文必须在多个交互或用户之间组装。像Caffeine这样的库或像Quarkus缓存扩展这样的框架可用于以低延迟存储和检索最近的LLM上下文片段。适当调优的缓存不仅改善响应时间，还减少冗余的计算和数据库查询。

### 短期上下文的热内存

**短期记忆**指的是会话中活跃的、最近的上下文，比如聊天中最后几条用户和助手的交流。这种记忆通常直接传递到模型的上下文窗口中，以维护跨对话的连续性。为了有效管理这一点，应用需要一个短生命周期的、会话范围的内存存储，可以快速更新和查询。

在基于Java的系统中，你可以使用内存存储、会话Bean或轻量级的键值存储如Redis。需要小心管理以避免超出模型的token限制；像总结或有选择地剪枝旧消息这样的策略有助于管理上下文相关性。

### 长期语义记忆的向量数据库

当需要在更长时间范围内存储和检索领域特定知识或用户级别的记忆时，传统的关系型数据库是不够的。**语义相似度**使得基于含义的检索变得更快。这就是向量数据库发挥作用的地方。

### 归档数据和大规模存储库的冷存储

并非所有信息都需要实时立即可访问。历史记录、日志、批量文档和很少访问的知识库可以存储在**冷存储**系统中，如Amazon S3或传统的关系型数据库。这些系统不会在每次请求的运行时被查询，但可能是定时摄入流水线的一部分，该流水线将数据加载、分块并索引到向量存储中。它们也可以在延迟不那么关键时支持临时的检索流程。冷存储的关键角色是为企业知识提供一个持久的、可扩展的真实数据源，可以在需要时转换为上下文资产。

### 组合存储层级以实现高效的上下文交付

一个健壮的上下文工程系统不依赖单一类型的存储。相反，它根据数据访问模式、延迟需求和任务复杂度在多个层级之间进行编排。**缓存**作为速度的第一线，**短期记忆**维护对话连续性，**向量存储**提供语义搜索，**冷存储**以持久知识支撑整个系统。这些组件一起确保正确的信息以正确的形式在正确的时间提供给LLM。

这种分层方法使开发者能够对成本、延迟和相关***进行细粒度控制，从而使AI应用既响应迅速又扎根于企业特定的知识。
